# Screeps Bot 架构设计

> 本文档记录本 bot 的核心设计理念、需求驱动模型,以及从「脚本」演进到「系统」的路线图。
> 适用环境:**私服**(CPU 近乎无限,缓存类优化可暂缓,但正确性设计不打折)。

---

## 1. 三条不可违背的核心理念

### 1.1 没有事件,只有轮询

Screeps 引擎是**每 tick 跑一遍**的轮询模型,**没有事件机制**:

- `Source` / `Structure` 等都是纯数据对象,**不会自己运行代码**,无法「主动发消息」。
- 函数 / 闭包**存不进 `Memory`**(只能存 JSON)。

所以「某建筑发出缺人通知」这种事件思维,在 Screeps 里**必须翻译成**:

> 我的代码每 tick **主动去扫描世界状态**,替建筑判断它缺什么。

「事件驱动(出事方通知)」和「轮询(关心方主动查)」是同一件事的两个方向,Screeps 只支持后者。

### 1.2 creep 无状态,状态存 Memory

为对抗引擎不定期的 **global reset**(JS 堆会被清空,只有 `Memory` 和实体存活):

- creep 本身不存逻辑,「它是谁、参数是什么」全部存在 `Memory.creepConfigs`。
- `Creep.prototype.work()` 每 tick 从 Memory 读配置 → 分发到对应角色行为。

### 1.3 队列每 tick 重建,绝不跨 tick 记忆

需求队列代表「**此刻缺什么**」,是时刻变化的数据,因此:

- **每 tick 新建空队列**,扫描世界填充,用完即弃。
- **绝不持久化到 Memory**。持久化会导致旧需求堆积 → 重复生产 → 脏数据。

带来的三大好处:**自愈、免去重、无需维护**。creep 死了,下 tick 扫描自动重新发现缺口并补上,不需要任何「死亡善后」逻辑。

> 判断数据该不该缓存的唯一标准:**会不会变**。
> 不变的(source 列表、地形)→ 可缓存复用;会变的(队列/当前需求)→ 每 tick 重建。

## 2. 需求驱动模型(Demand-Driven)

### 2.1 控制反转

不再由 `main.ts` 命令式地写死「造 3 个 harvester」,而是:

```
每 tick:
  1. 扫描世界状态 → 各「需求源」产生需求(纯数据)
  2. 消费者按优先级处理需求
```

加新角色 = 加一个扫描函数 + 在 main 里加一行,**消费者逻辑一个字不用改**。

### 2.2 两类需求,两类消费者

这是本架构最关键的抽象:**需求是通用概念,但不同需求的消费者不同。spawn 只是消费者之一。**

| | 生产需求 | 任务需求 |
|---|---|---|
| 含义 | 缺一个**还不存在**的 creep | 要一个**已存在**的 creep 去干事 |
| 消费者 | **spawn** | **现有的空闲 creep** |
| 产出 | 一个新 creep | 一个 creep 被派去干活 |
| 例子 | 没人挖矿 → 造矿工 | 有工地 → builder 去建;墙破 → repairer 去修 |

两类需求**共用同一套引擎**:`扫描 → 产生需求 → 按优先级消费 → tick 内出队 → tick 重建`,区别只在「消费者池」是谁。

### 2.3 请求的数据结构

请求必须可序列化(纯数据):

```ts
interface SpawnRequest {
  role: keyof RoleType;
  priority: number;            // 数字越小越优先
  bodyTemplate: BodyPartConstant[];
  memory: Record<string, any>; // 给 creep 的初始配置(含目的地)
}
```

**粒度原则:一条请求 = 一个单位需求。** source A 缺 1 个矿工就 push 1 条;controller 缺 2 个 upgrader 就 push 2 条。队列里有几条,就是还差几个 creep——这样多消费者各取一条,天然不会超量。

### 2.4 优先级仲裁

队列按 `priority` 升序,**「造不起最紧急的就停,绝不退而求其次」**:

```
采矿(5) < 静态矿工修容器 < 建造(20) < 升级(30)
```

若最高优先级的请求当前能量造不起,**宁可等攒满,也不跳过去造低优先级**,否则永远攒不够钱造关键大 creep。

### 2.5 多消费者:tick 内出队

单 spawn 时,挑一个造完即停,无需出队(一个 spawn 一 tick 只能孵化一个)。

多 spawn(RCL7 第二 spawn、RCL8 第三 spawn)时,**同一 tick 内**多个消费者要协调:

```
queue 按优先级排序
for spawn in spawns:
    请求 = 队列里第一个造得起的
    spawn.spawnCreep(请求)
    queue.remove(请求)   # ← tick 内出队,下个 spawn 挑不到它了
```

> ⚠️ 「tick 内出队」是**同一 tick 内多消费者的协调手段**,与「跨 tick 记忆」是两回事。
> 出队照常做,但 tick 结束整张队列照样作废、下 tick 重建。两者并存不矛盾。

## 3. creep 的两种绑定模式

| | 专用 creep | 通用 creep |
|---|---|---|
| 例子 | **静态矿工**(绑定 source) | hauler、builder、万能 worker |
| 绑定时机 | **出生即绑定**,终生服务一个目标 | 不绑定,每 tick 临时领最紧急任务 |
| 去哪干活 | 直奔出生时写入 memory 的目标 | 每 tick 自己扫世界 / 领任务 |

### 3.1 专用 creep:出生即绑定,马上上岗

触发生产的需求**本身就带着目的地**(「container A 缺人」)。因此:

- 在 `spawnCreep` 时就把 `sourceId` / `containerId` 写进 creep 的 memory。
- creep 一出生就知道自己是谁的,**直奔目标,不等任何任务队列分派**。

**不要**让它「出生后等任务队列通知」——那是信息浪费,还可能分错、产生竞态。

### 3.2 出生绑定顺带解决「重复生产」

从「spawn 开始孵化」到「creep 走到岗位」有个时间窗口,期间岗位上其实还没人干活。若扫描只看「岗位上站没站人」,会误判仍缺人 → 重复生产。

解法:出生时把 `creepName` 登记进「坑位表」,扫描判断的是 **「这个坑的 creepName 还活着吗」**(孵化中的 creep 也在 `Game.creeps` 里),于是窗口期内坑位已被「预留」,不会重复造。

```
写 creepName 到坑位 ┬→ 告诉 creep:你是 A 矿的(目的地)
                    └→ 标记坑位:已预留(防重复生产)
```

## 4. 任务的「独占预留表」用在哪

`{ targetId, creepName }` 这种指派对象的本质作用是**独占预留**——「这个目标已被认领,别人别来」。

| | 需要预留表吗 | 原因 |
|---|---|---|
| 静态矿工 ↔ container | ✅ 需要 | 1:1 永久绑定,多派人是浪费,必须防重复 |
| builder ↔ 工地 | ❌ 多数不需要 | 工地建完即消失,多 builder 同建一个反而加速;工地本身就是引擎帮你存的持久需求 |
| repairer ↔ 破损建筑 | ❌ 不需要 | 同上,破损状态摆在那,自取即可 |

**原则:需要「独占」的任务才用预留表;可共享、自消失的任务让 creep 每 tick 自己扫世界自取。**

> 要统一的是「需求 → 消费者」这个**模式**,不是「所有任务都套同一种数据结构」。
> 不要为了形式统一,给 builder 强行套独占表,否则会陷进「工地建完了表里还指着它」「俩 builder 抢登记」的无谓维护。

## 5. 结构的「需求画像」

判断一个结构怎么接入系统,只问三件事:**要不要建?会不会坏?要不要喂?**

| 结构 | 建造(一次性) | 衰减 → 修理 | 补充能量(持续) | 备注 |
|---|---|---|---|---|
| road | ✅ | ✅ 100hits/1000tick | — | 需规划路线 |
| container | ✅ | ✅ 5000hits/500tick(自有房) | — | 静态矿工可自修脚下容器 |
| extension | ✅(RCL 限量) | ❌ 不衰减 | ✅ 必须持续填能量 | 升 RCL 自动开闸建更多 |
| spawn | (已有) | ❌ | ✅ | |
| tower | ✅(RCL) | ❌ | ✅ | |
| rampart | ✅ | ✅ 300hits/100tick | — | |
| wall | ✅ | ❌ | — | |

**会衰减的只有三种:road / container / rampart**,它们才进「修理」线。

一个结构可能同时挂在多条线上,例如 extension =「要建(RCL 限量)+ 不坏 + 要喂」。

### 5.1 衰减是动态的:不能假设结构永远存在

container 不修会在自有房约 25000 tick 后消失(无主房快 5 倍)。处理方式纳入需求模型:

| container 状态 | 产生的需求 | 消费者 |
|---|---|---|
| 不存在 | 建造需求:放 container 工地 | builder |
| 存在但没人挖 | 生产需求:造静态矿工 | spawn |
| 血量低 | 修理需求:矿工自修 / repairer | 矿工自己 / repairer |

**最优雅方案:静态矿工边挖边自修脚下 container**(它本就站在上面,且手握能量,1 个 WORK 修 100hits 只花 1 能量,抵消 ~10hits/tick 的衰减零成本)。这样 container 永不消失,善后清理逻辑基本用不上。

## 6. 铺路与 extension 的统一

两者都**没有专属角色**,都拆成「规划 + 建造」:

```
铺路:     偶尔用 PathFinder 算路线(spawn↔source↔controller) → 放 road 工地 → 汇入建造线
                                                              → 路会衰减 → 汇入修理线
extension: 规划网格位置(buildStructs.ts) → 放 extension 工地 → 汇入建造线
                                          → 不衰减,但要填能量 → 汇入补充线
```

**规划步骤要少做**(布局基本不变,算一次即可,甚至存 Memory),绝不每 tick 算。放完工地后,一切复用通用的建造 / 修理 / 补充流水线。

## 7. 端到端样板:静态采矿链路

这是最能体现完整架构的一条线(独占 + 出生绑定 + 防重复 + 自维护):

```
扫描坑位表(containerForSuperHarvest)
  └─ 坑位 container 不存在 → 建造需求 → builder 建 container
  └─ 坑位无活人(creepName 失效) → 生产需求(带 sourceId/containerId) → spawn 消费
        └─ 矿工出生即绑定 → prepare 阶段走到 container → 站上去
              └─ 边挖边自修脚下 container(防衰减消失)
              └─ creepName 登记进坑位(防重复生产)
  └─ 矿工老死 → 坑位空 → 下 tick 扫描自动重新发"生产需求" → 补新矿工(自愈)
```

全程无跨 tick 脆弱状态,矿工绝不会因「忘了补」而长期断档。

## 8. body 动态生成

不要写死 `[WORK, CARRY, MOVE]`(等于永远按最穷时打仗)。按可用能量重复模板单元:

```ts
function repeatBody(unit, energy, maxUnits): BodyPartConstant[]
// 用 room.energyCapacityAvailable 造最强体(平时);
// 用 room.energyAvailable 造当下买得起的(紧急救场,防 creep 全灭锁死)
```

**例外:静态矿工固定 `[WORK×5, CARRY, MOVE]`,不参与扩容**——source 每 tick 只产 10 能量,5 个 WORK 正好榨干,多堆 WORK 是浪费。

## 9. 缓存策略(私服暂缓,记录原则)

私服 CPU 近乎无限,**当前阶段缓存可不做**。待上正式服或多房间时再补:

- **可缓存**:`FIND_SOURCES`、地形等**永不变**的数据,跨 tick 复用。
- **绝不缓存**:队列 / 当前需求等**每 tick 变**的数据。
- 缓存的是「造队列用的不变原料」,不是队列本身(队列是输出,每 tick 现做)。

类比:缓存 = 冰箱里的食材(买一次反复用);队列 = 这顿现炒的菜(不留剩菜当新菜)。

## 10. 演进路线图

| 阶段 | 该补的能力 | 对应游戏目标 |
|---|---|---|
| → RCL3 | 动态 body、container 静态采矿成型 | 提速、第一座 tower |
| → RCL4 | (CPU 紧张才做)缓存层、tower 自动防御/修理 | Storage,物流分水岭 |
| → RCL5 | 物流 / 任务系统(围绕 Storage 的 hauler)、专职 filler | Link 网络 |
| RCL6+ | 市场交易、矿物链、多房间扩张 | 工业化、规模化 |

**推进顺序(性价比从高到低):动态 body(小) → spawn 请求队列(中) → 物流任务系统(大)**,层层递进。

## 附:核心循环全景

```
                          ┌─ 生产需求 ──→ spawn 消费 ──→ 造新 creep(专用则出生绑定)
扫描世界状态 ──────────────┤
(每 tick,无事件,主动轮询)  └─ 任务需求 ──→ 现有 creep 消费 ──→ 干活
                              ├─ 建造:工地(引擎托管) ← builder 自取
                              ├─ 修理:破损建筑(状态摆在那) ← repairer 自取
                              └─ 补充:空的 extension/spawn/tower ← 搬运 creep 自取

铺路 / extension = 偶尔规划 → 放工地 → 汇入上面对应的需求线
队列:每 tick 重建、不跨 tick 记忆、自愈;多消费者 tick 内出队协调
```
